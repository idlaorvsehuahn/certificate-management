# PKI Concepts Explained in Context

Rather than learning each PKI term in isolation, it's much easier to understand them by following the lifecycle of a certificate—from the moment a company wants HTTPS on its website until your browser establishes a secure TLS connection.

---

# Big Picture

Imagine your company owns:

```text
https://api.mycompany.com
```

You want browsers to trust your server.

The complete PKI ecosystem looks like this:

```text
                Root CA
                   │
          Signs Intermediate CA
                   │
         Intermediate CA
                   │
      Signs Server Certificate
                   │
      api.mycompany.com Certificate
                   │
             Your Web Server
                   │
            Browser verifies
                   │
           Secure TLS Connection
```

Now let's understand every component involved.

---

# 1. X.509

**X.509 is the standard format for digital certificates.**

Think of it as the blueprint that defines how certificates are structured, just like:

* PDF defines document format.
* JPEG defines image format.
* X.509 defines certificate format.

An X.509 certificate typically contains information such as:

```text
Version: 3

Serial Number:
123456789

Subject:
api.mycompany.com

Issuer:
Let's Encrypt

Public Key:
ABC123...

Valid From:
Jan 1, 2026

Valid To:
Apr 1, 2026

Digital Signature:
...
```

Almost every HTTPS certificate on the internet follows the X.509 standard.

---

# 2. Public Key vs Private Key

A public key and private key are generated together as a cryptographic key pair.

```text
Private Key  <------ mathematically linked ------> Public Key
```

## Private Key

The private key **never leaves your server**.

It is used for:

* Creating digital signatures
* Proving ownership of the certificate
* Decrypting data (for algorithms that support it)
* Authenticating the server during TLS

If someone steals your private key, they can impersonate your server.

---

## Public Key

The public key is safe to share.

It is embedded inside your certificate so that anyone connecting to your server can access it.

Knowing the public key **does not reveal the private key**.

### Simple analogy

Imagine a mailbox.

Public key:

```text
Anyone can drop letters inside.
```

Private key:

```text
Only the owner can open it.
```

---

# 3. Certificate Authority (CA)

A Certificate Authority (CA) is a trusted organization responsible for verifying identities and issuing certificates.

Examples include:

* Let's Encrypt
* DigiCert
* Sectigo

Think of a CA as a passport office.

Without a CA, anyone could create a certificate claiming:

```text
I'm google.com
```

The CA verifies that the requester actually owns the domain before issuing the certificate.

---

# 4. CSR (Certificate Signing Request)

Suppose you're deploying:

```text
api.mycompany.com
```

The first step is generating a key pair:

```text
Private Key

Public Key
```

Next, you create a **Certificate Signing Request (CSR)**.

A CSR contains:

```text
Domain Name

Organization

Country

Public Key
```

**Important:** A CSR never contains the private key.

Example:

```text
CSR

Subject:
api.mycompany.com

Public Key:
ABC123...
```

The CSR is sent to the Certificate Authority.

The CA verifies ownership of the domain.

If everything checks out, the CA issues a certificate containing the **same public key** that was included in the CSR.

---

# 5. Subject

The **Subject** identifies **who owns the certificate**.

Example:

```text
Subject

CN = api.mycompany.com

Organization = MyCompany

Country = IN
```

Think of it as the identity printed on an ID card.

---

# 6. Issuer

The **Issuer** identifies **who issued the certificate**.

Example:

```text
Issuer

Let's Encrypt
```

So the certificate effectively says:

```text
Subject:
api.mycompany.com

Issued By:
Let's Encrypt
```

---

# 7. SAN (Subject Alternative Name)

Modern certificates can secure multiple domains using the **Subject Alternative Name (SAN)** extension.

Example:

```text
Certificate

SAN

example.com

www.example.com

api.example.com

admin.example.com
```

When you visit:

```text
https://api.example.com
```

your browser checks whether:

```text
api.example.com
```

appears in the SAN list.

If it doesn't, the TLS connection is rejected.

> **Note:** Modern browsers validate hostnames using the SAN extension. The older Common Name (CN) field is no longer used for hostname verification.

---

# 8. Expiration

Certificates have a limited lifetime.

Example:

```text
Valid From

Jan 1, 2026

Valid To

Apr 1, 2026
```

Certificates expire because:

* Stolen keys shouldn't remain valid forever.
* Organizations should rotate keys regularly.
* Security standards evolve over time.

Browsers reject expired certificates.

---

# 9. Root CA

How does the browser trust a Certificate Authority?

Browsers and operating systems ship with a built-in list of **trusted Root Certificates**.

Examples:

```text
Windows Trust Store

Apple Trust Store

Linux Trust Store

Firefox Trust Store
```

These trusted root certificates are installed with your operating system or browser.

Because the browser already trusts the Root CA, it can trust certificates that ultimately trace back to it.

---

# 10. Intermediate CA

Root CAs almost never issue certificates directly.

Instead, the hierarchy looks like this:

```text
Root CA

↓

Signs

↓

Intermediate CA

↓

Signs

↓

Your Certificate
```

### Why?

The Root CA's private key is extremely valuable.

To minimize risk:

* The Root CA remains offline and highly protected.
* Intermediate CAs perform the day-to-day certificate issuance.

If an Intermediate CA is compromised, it can be revoked without replacing the Root CA.

---

# 11. Certificate Chain

When your server sends its certificate, the browser must verify that it can trust it.

It builds a **certificate chain**:

```text
Server Certificate

↓

Signed by Intermediate CA

↓

Signed by Root CA

↓

Root CA already trusted
```

Think of it like professional references:

```text
Employee

↓

Recommended by Manager

↓

Recommended by CEO

↓

CEO already trusted
```

The browser verifies every digital signature in the chain until it reaches a trusted Root CA.

If any verification fails, the certificate is rejected.

---

# Putting Everything Together

Suppose you own:

```text
api.mycompany.com
```

### Step 1 — Generate a key pair

```text
Private Key

Public Key
```

---

### Step 2 — Create a CSR

```text
CSR

Subject:
api.mycompany.com

Public Key
```

---

### Step 3 — Send the CSR to a Certificate Authority

---

### Step 4 — The CA verifies ownership of the domain

---

### Step 5 — The CA issues an X.509 certificate

```text
Subject:
api.mycompany.com

Issuer:
Let's Encrypt

Public Key:
ABC123

SAN:
api.mycompany.com

Expiration:
Apr 1, 2026
```

---

### Step 6 — The certificate is signed by an Intermediate CA

```text
Intermediate CA

↓

Signs

↓

Your Certificate
```

---

### Step 7 — A browser connects

The server sends:

* Its X.509 certificate
* Intermediate certificate(s)

---

### Step 8 — The browser builds the certificate chain

```text
Server Certificate

↓

Intermediate CA

↓

Root CA
```

---

### Step 9 — The browser verifies

* Is the certificate expired?
* Does the requested hostname exist in the SAN?
* Is the issuer valid?
* Is every digital signature in the chain valid?
* Does the chain terminate at a trusted Root CA?

If all checks pass, the browser trusts the server's public key.

TLS can now proceed (typically using ECDHE) to establish a shared session key, after which all HTTP traffic is encrypted using a fast symmetric cipher such as AES-GCM or ChaCha20-Poly1305.

---

# One-Page Mental Model

```text
                 Root CA
             (Trusted by Browser)
                     │
          Signs Intermediate CA
                     │
          Intermediate CA
                     │
          Signs Certificate
                     │
      +----------------------------------+
      |          X.509 Certificate       |
      |----------------------------------|
      | Subject: api.mycompany.com       |
      | Issuer: Let's Encrypt            |
      | Public Key                       |
      | SAN: api.mycompany.com           |
      | Expiration: Apr 1, 2026          |
      +----------------------------------+
                     │
          Installed on Server
                     │
          Browser Downloads It
                     │
       Browser Verifies Certificate Chain
                     │
       Browser Trusts Public Key
                     │
      TLS (ECDHE) Creates Shared Secret
                     │
     AES-GCM / ChaCha20 Encrypt HTTP Traffic
```

## Summary

PKI is the trust infrastructure behind HTTPS. It enables browsers to verify that they are communicating with the legitimate server before establishing an encrypted connection.

Understanding how X.509 certificates, public/private keys, CSRs, Certificate Authorities, Root CAs, Intermediate CAs, certificate chains, SANs, issuers, subjects, and expiration all fit together provides the foundation for understanding modern TLS.
