# Understanding TLS: Handshake, Session Keys, Certificate Verification, Trust Chain, and mTLS

Once you understand PKI, the next step is understanding how TLS actually uses it to establish a secure connection.

Think of PKI as **the system that establishes trust**, while TLS is **the protocol that uses that trust to communicate securely**.

Let's follow a real HTTPS request from beginning to end.

---

# Big Picture

Imagine you visit:

```text
https://api.mycompany.com
```

The entire flow looks like this:

```text
Browser
    │
    │ 1. TLS Handshake
    ▼
Server sends Certificate
    │
    │ 2. Browser verifies Certificate
    ▼
Browser builds Trust Chain
    │
    │ 3. Identity verified
    ▼
ECDHE creates Shared Secret
    │
    │ 4. Session Keys derived
    ▼
AES-GCM encrypts HTTP traffic
```

Everything before the HTTP request is called the **TLS Handshake**.

---

# 1. TLS Handshake

The TLS Handshake is the process by which the client and server:

* Verify each other's identity (server always, client only in mTLS)
* Agree on cryptographic algorithms
* Perform key exchange
* Derive session keys
* Begin encrypted communication

Think of it as two strangers meeting before having a private conversation.

Instead of immediately exchanging secrets, they first answer questions like:

```text
Who are you?

Can I trust you?

Which encryption algorithms do you support?

Can we agree on a shared secret?
```

Only after all of those steps succeed does encrypted communication begin.

---

## TLS 1.3 Handshake Flow

### Step 1 — ClientHello

The browser sends:

```text
TLS Version Supported

Cipher Suites

Client Random

ECDHE Public Key

Extensions (SNI, ALPN, etc.)
```

Notice that the browser already generates an **ephemeral ECDHE key pair** before sending the first message.

---

### Step 2 — ServerHello

The server responds with:

```text
Chosen TLS Version

Chosen Cipher Suite

Server Random

ECDHE Public Key

Server Certificate

Certificate Chain

Digital Signature
```

The server also signs the handshake parameters using the private key corresponding to its certificate.

This proves that the ephemeral public key actually belongs to the legitimate server.

---

### Step 3 — Certificate Verification

The browser now validates the certificate.

We'll discuss this process in detail shortly.

---

### Step 4 — Shared Secret

Using ECDHE:

The browser combines:

```text
Browser Private Key

+

Server Public Key
```

The server combines:

```text
Server Private Key (ECDHE)

+

Browser Public Key
```

Both independently compute the exact same shared secret.

Neither side ever transmits that secret across the network.

---

### Step 5 — Session Keys

The shared secret is **not** used directly for encryption.

Instead, TLS combines:

* Shared Secret
* Client Random
* Server Random

and feeds them into a Key Derivation Function (KDF).

The KDF generates multiple independent session keys.

---

### Step 6 — Secure Communication Begins

All HTTP requests and responses are now encrypted using fast symmetric encryption.

Typically:

* AES-GCM
* ChaCha20-Poly1305

The TLS Handshake is complete.

---

# 2. Session Keys

One of the most misunderstood concepts in TLS is the difference between the **shared secret** and the **session keys**.

They are **not the same thing**.

The flow looks like this:

```text
ECDHE

↓

Shared Secret

↓

Key Derivation Function (HKDF)

↓

Session Keys

↓

AES-GCM Encryption
```

TLS derives multiple keys instead of using one key everywhere.

For example:

```text
Client → Server Encryption Key

Server → Client Encryption Key

Initialization Values

Other protocol-specific secrets
```

Using separate keys prevents one key from being reused for multiple purposes.

This is called **key separation**, and it improves security.

---

# 3. Why Asymmetric Encryption Isn't Used for All Traffic

A common question is:

> If public-key cryptography is so secure, why not encrypt every HTTP request with it?

The answer is performance.

---

## Symmetric Encryption

Algorithms like AES require only one shared key.

Advantages:

* Extremely fast
* Hardware accelerated on modern CPUs
* Efficient for large amounts of data

Encrypting gigabytes of traffic with AES is practical.

---

## Asymmetric Encryption

Algorithms like RSA or ECDSA (used for signatures) involve much more computationally expensive mathematical operations.

Asymmetric cryptography is excellent for:

* Authentication
* Digital signatures
* Key exchange (using ECDHE)

It is **not** efficient for encrypting continuous application data.

---

Imagine downloading a 5 GB movie.

Using RSA or similar public-key operations for every packet would be prohibitively slow.

Instead, TLS uses:

```text
Asymmetric Cryptography

↓

Authenticate

↓

Exchange Secret

↓

Symmetric Encryption

↓

Encrypt Everything
```

This combination provides both strong security and excellent performance.

---

# 4. Certificate Verification

When the browser receives the server's certificate, it doesn't simply accept it.

It performs multiple checks.

---

## Check 1 — Is the Certificate Expired?

Example:

```text
Valid From

Jan 1, 2026

Valid To

Apr 1, 2026
```

If today's date falls outside that range,

the browser rejects the certificate.

---

## Check 2 — Does the Hostname Match?

Suppose you visit:

```text
https://api.mycompany.com
```

The browser checks the SAN extension.

Example:

```text
SAN

api.mycompany.com

www.mycompany.com
```

If the requested hostname isn't present,

the handshake fails.

---

## Check 3 — Is the Digital Signature Valid?

The CA signs the certificate using its private key.

The browser verifies that signature using the CA's public key.

If the certificate was modified,

the signature verification fails.

---

## Check 4 — Is the Issuer Trusted?

The browser checks whether the issuer can be linked back to a trusted Root CA.

This leads us to the trust chain.

---

# 5. Trust Chain

The browser never blindly trusts a server certificate.

Instead, it builds a chain of trust.

Example:

```text
Server Certificate

↓

Signed by Intermediate CA

↓

Signed by Root CA

↓

Root Certificate already trusted
```

The browser verifies every signature in that chain.

If every link is valid,

the server is trusted.

If any signature fails,

TLS immediately terminates the connection.

---

# Putting the Entire TLS Flow Together

```text
Browser                              Server

ClientHello ----------------------->

                         ServerHello
                         Certificate
                         Certificate Chain
                         Signature
<-----------------------------------

Verify Certificate

Verify Trust Chain

Compute Shared Secret

                         Compute Shared Secret

Derive Session Keys

Encrypted HTTP Begins
```

At this point, all HTTP traffic is protected using symmetric encryption.

---

# What is mTLS (Mutual TLS)?

In normal HTTPS,

only the **server proves its identity**.

```text
Browser

↓

Verifies

↓

Server Certificate
```

The server has no cryptographic proof of who the client is.

Authentication usually happens later using:

* Username/password
* JWT
* OAuth
* Session cookies

---

## Mutual TLS (mTLS)

In mTLS,

**both sides authenticate each other using certificates.**

```text
Client

↓

Certificate

↓

Server

AND

Server

↓

Certificate

↓

Client
```

Both participants prove ownership of their private keys.

This is why it's called **Mutual TLS**.

---

# mTLS Handshake

The beginning is identical to normal TLS.

### Step 1

The client sends:

```text
ClientHello
```

---

### Step 2

The server replies:

```text
ServerHello

Certificate

Certificate Chain
```

---

### Step 3

The server requests a client certificate.

```text
CertificateRequest
```

This is the major difference from standard TLS.

---

### Step 4

The client sends:

```text
Client Certificate

Certificate Chain
```

---

### Step 5

The client proves ownership of the certificate's private key by digitally signing parts of the handshake.

The server verifies that signature using the client's public key.

Possessing the certificate alone isn't enough—the client must also demonstrate control of the corresponding private key.

---

### Step 6

The server validates:

* Client certificate
* Certificate chain
* Trusted issuer
* Expiration
* Digital signature
* Any application-specific authorization rules

If everything is valid,

the handshake completes.

---

# Normal TLS vs mTLS

| Feature                     | TLS                   | mTLS                                                                 |
| --------------------------- | --------------------- | -------------------------------------------------------------------- |
| Server authenticated        | ✅                     | ✅                                                                    |
| Client authenticated        | ❌                     | ✅                                                                    |
| Server certificate required | ✅                     | ✅                                                                    |
| Client certificate required | ❌                     | ✅                                                                    |
| Browser support             | Excellent             | Less common for public websites                                      |
| Typical use                 | Websites, public APIs | Internal services, enterprise APIs, service-to-service communication |

---

# Where is mTLS Used?

mTLS is commonly used where strong machine identity is required.

Examples include:

### Microservices

```text
Payment Service

↓

mTLS

↓

Order Service
```

Each service verifies that it's communicating with an authorized service.

---

### Kubernetes

Pods communicate securely using mTLS in service meshes such as Istio or Linkerd.

---

### Banking Systems

Banks authenticate both ends of sensitive API connections using client certificates.

---

### Enterprise APIs

Organizations often issue certificates to approved customers or partners.

Only clients with trusted certificates can connect.

## Summary

TLS combines **authentication**, **key exchange**, and **symmetric encryption** into a secure protocol for protecting network traffic.

* **PKI** establishes trust through certificates.
* **TLS Handshake** verifies identity and negotiates cryptographic parameters.
* **ECDHE** creates a shared secret.
* **HKDF** derives independent session keys.
* **AES-GCM or ChaCha20-Poly1305** encrypt application data efficiently.
* **Certificate verification** and the **trust chain** ensure you're talking to the legitimate server.
* **mTLS** extends TLS by requiring both the client and server to authenticate using certificates, making it ideal for service-to-service communication and other high-security environments.
