# Improve Certificate Issuance UX to Follow Real PKI Practices

The current implementation displays the full X.509 PEM certificate immediately after issuance and also shows the PEM again on the certificate details page.

I want to redesign this behavior to better reflect how real Certificate Management Platforms operate.

The goal is to distinguish between **public certificate information** and **sensitive issuance artifacts**, while providing a better user experience.

---

# Desired User Flow

## Certificate Issuance

After successfully issuing a certificate, the user should be redirected to a success page.

This page represents the **only issuance response**.

It should display:

* Success message
* Certificate ID
* Certificate PEM
* Private Key PEM (see below)
* Copy buttons
* Download buttons
* Security warning

The page should make it clear that the private key is only available once.

Example warning:

> This private key is displayed only once.
> Please copy or download it now.
> It cannot be retrieved again after leaving this page.

The warning should be visually prominent.

---

# API Response

Modify the Certificate Service response to better represent a certificate package.

Instead of only returning:

```json
{
    "pem": "..."
}
```

Return something similar to:

```json
{
    "certificate": {
        "id": "...",
        "subject": "...",
        "issuer": "...",
        "serial_number": "...",
        "issued_at": "...",
        "expiration": "...",
        "status": "ACTIVE",
        "san_dns_names": [...]
    },
    "certificate_pem": "...",
    "private_key_pem": "...",
    "warning": "The private key is displayed only once."
}
```

If the backend does not yet generate a private key separately, create the response structure now and explain how it would work in a production implementation.

---

# Success Page

The issuance success page should contain:

✔ Certificate successfully issued

Certificate ID

Certificate PEM

Private Key PEM

Copy Certificate button

Copy Private Key button

Download Certificate button

Download Private Key button

Return to Inventory button

Display a security warning explaining that the private key will never be shown again.

---

# Inventory Page

The Inventory page should **not** display PEM data.

Display only metadata such as:

* Subject
* Issuer
* Status
* Expiration
* Created Date

This page is intended to act as a searchable inventory.

---

# Certificate Details Page

The Details page should **not** display the raw PEM certificate.

Instead, display parsed certificate information in a readable format.

Examples:

Certificate ID

Subject

Issuer

Status

Serial Number

Issued At

Expiration

Signature Algorithm

Public Key Algorithm

Subject Alternative Names

SHA-256 Fingerprint

Validity

This page should look like a certificate inspection page rather than a PEM viewer.

---

# Certificate Parser

The backend already contains a certificate parser (or will soon).

Use the parsed certificate information to populate the Details page.

Avoid simply rendering database fields whenever possible.

This demonstrates the certificate parser bonus feature.

---

# Download Actions

Instead of rendering a huge PEM block everywhere, provide actions such as:

Download Certificate (.pem)

Copy Certificate

Inspect Certificate

If desired, add a "View Raw PEM" accordion that is collapsed by default.

The parsed view should be the primary experience.

---

# Security Goals

The frontend should clearly communicate the difference between:

Public Certificate

Private Key

The private key is sensitive.

The certificate is not.

Even if the backend currently stores only the certificate PEM, structure the UI so that it could easily support a real PKI implementation later.

---

# Future Compatibility

The design should make it easy to later support:

PKCS#12 download

Certificate chain download

Intermediate certificates

Certificate renewal

Certificate revocation

without redesigning the UI.

---

# Goal

The final experience should resemble a professional Certificate Management Platform instead of a CRUD application.

It should educate users about certificate handling while following common PKI workflows used by enterprise certificate authorities.
