# Refactor Docker Deployment Structure

I want to refactor the Docker deployment structure to follow production-oriented microservice practices.

Do **not** change the application logic.

Only reorganize the Docker deployment, PostgreSQL initialization, and startup flow.

---

# Current Situation

Currently:

* `docker-compose.yml` exists at the workspace root.
* PostgreSQL is started from Docker Compose.
* NATS is started from Docker Compose.
* `POSTGRES_DB` is used to automatically create `certificate_db`.
* `init.sql` creates only `inventory_db`.

This creates inconsistent database initialization.

I want a cleaner and more maintainable approach.

---

# Desired Startup Flow

The desired startup sequence should be:

```text
docker compose up

↓

PostgreSQL starts

↓

PostgreSQL executes initialization scripts

↓

certificate_db created

inventory_db created

↓

Certificate Service starts

↓

Runs SQLx migrations

↓

Creates certificate tables

↓

Inventory Service starts

↓

Runs SQLx migrations

↓

Creates inventory tables

↓

System Ready
```

Database creation and schema creation must be treated as separate responsibilities.

# docker-compose.yml

Keep `docker-compose.yml` at the repository root.

Reason:

Docker Compose orchestrates the **entire platform**, not a single service.

It should start:

* PostgreSQL
* NATS

(and later, optionally, the microservices)

Developers should simply execute:

```bash
docker compose up
```

from the repository root.

---

# PostgreSQL Initialization

Do **not** rely on:

```yaml
POSTGRES_DB
```

to create one database.

Instead:

Configure only:

```yaml
POSTGRES_USER
POSTGRES_PASSWORD
```

Mount:

```text
deploy/docker/postgres/init.sql
```

to:

```text
/docker-entrypoint-initdb.d/init.sql
```

The initialization script should create every required database.

Example:

```sql
CREATE DATABASE certificate_db;
CREATE DATABASE inventory_db;
```

Do not create tables.

Do not create indexes.

Do not insert seed data.

Its only responsibility is creating databases.

---

# SQLx Migrations

Each service owns its own schema.

Certificate Service:

```text
services/
    certificate-service/
        migrations/
```

Inventory Service:

```text
services/
    inventory-service/
        migrations/
```

Each service should automatically execute its own SQLx migrations during startup.

Responsibilities:

Certificate Service:

* certificate tables
* indexes
* constraints

Inventory Service:

* inventory tables
* indexes
* constraints

Never place schema creation inside init.sql.

---

# Responsibilities

PostgreSQL init.sql

Responsible for:

* creating databases

Not responsible for:

* tables
* schema
* indexes
* seed data

---

SQLx migrations

Responsible for:

* schema creation
* schema evolution
* indexes
* constraints
* future schema changes

---

Docker Compose

Responsible for:

* infrastructure orchestration
* networking
* volumes
* service startup

Not responsible for database schema.

---

# Certificates

Development certificates generated by `tools/certgen` should be written into:

```text
deploy/certs/
```

The output directory should be configurable.

Example:

```bash
cargo run -p certgen -- --output ./deploy/certs
```

Avoid hardcoded paths.

---

# NATS

Docker Compose should also start NATS.

No additional configuration is required for now.

Services will connect through environment variables.

---

# Desired Result

The final architecture should clearly separate infrastructure provisioning from application schema management.

Infrastructure:

* Docker Compose
* PostgreSQL
* NATS
* Networks
* Volumes

Database Provisioning:

* init.sql

Application Schema:

* SQLx migrations owned by each service

Business Logic:

* Rust services

This separation should make the project easier to maintain and closer to production-grade backend systems.

---

# Implementation Notes

Before making any changes:

1. Explain what will change.
2. Explain why the change is beneficial.
3. Refactor incrementally.
4. Preserve existing functionality.
5. Do not move SQLx migrations into init.sql.
6. Do not rely on POSTGRES_DB to create one database while SQL creates another.
7. Ensure both databases are created consistently by PostgreSQL initialization.
