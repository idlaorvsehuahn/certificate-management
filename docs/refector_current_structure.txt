# Refactor the Repository into a Cargo Workspace

The current project consists of multiple independent Cargo projects:

* certificate-service
* inventory-service
* certgen

I want to refactor the repository into a production-quality Rust workspace similar to what a backend team would use for a microservice platform.

Do **not** rewrite business logic.

Focus only on reorganizing the project structure while preserving functionality.

# Workspace

Create a Cargo workspace.

The root Cargo.toml should contain workspace members.

Example:

```
members = [
    "services/certificate-service",
    "services/inventory-service",
    "crates/ca-engine",
    "crates/shared",
    "tools/certgen"
]
```

Use dependency inheritance where appropriate.

# Purpose of Each Module

## services/

Contains deployable applications.

Each service should contain:

* API
* Service layer
* Repository layer
* Messaging
* Configuration
* Telemetry

Services should not contain reusable cryptographic logic.

---

## crates/ca-engine

Create a new library crate.

This crate will eventually contain all Certificate Authority-related functionality.

Initially, it may contain only placeholder modules if necessary.

Suggested modules:

```
lib.rs

authority.rs

builder.rs

signer.rs

parser.rs

pem.rs

keys.rs

errors.rs

types.rs
```

The Certificate Service will depend on this library.

This crate should **not** depend on Axum, SQLx, NATS, or HTTP.

It should only contain cryptographic and X.509 related logic.

---

## crates/shared

Create another library crate.

This crate should contain reusable code shared across services.

Examples:

```
config/

errors/

events/

dto/

constants/

types/

telemetry/
```

Examples of shared objects:

CertificateIssuedEvent

CertificateRevokedEvent

Pagination

Shared error types

Configuration structs

Utility functions

Do not place service-specific business logic here.

---

## tools/certgen

The current certgen project should become a development tool.

Its responsibility is only:

Generate local development certificates.

It should generate:

Root CA

Certificate Service certificate

Inventory Service certificate

These certificates are only for:

HTTPS

mTLS

local development

Do not confuse these with business certificates issued by the Certificate Service.

# Certificates Directory

Move

```
certs/
```

to

```

```

The generated certificates should be placed here.

The output directory should be configurable through a command-line argument.

Example:

```
cargo run -p certgen -- --output ./deploy/certs
```

Avoid hardcoding paths.

# Architecture Principles

The repository should clearly separate:

Applications

Libraries

Development tools

Deployment artifacts

Documentation

Avoid mixing these concerns.

---

# Dependency Direction

Desired dependency graph:

```
Certificate Service
        │
        ▼
   ca-engine

Certificate Service
        │
        ▼
     shared

Inventory Service
        │
        ▼
     shared

certgen

(no dependency on services)
```

Services may depend on libraries.

Libraries must never depend on services.

The certgen tool should never depend on the services.

---

# Future Architecture

The long-term design should be:

```
services/

    certificate-service
            │
            ▼
        ca-engine
            │
            ▼
        shared

inventory-service
        │
        ▼
      shared
```

Later, the Certificate Service will use the CA Engine to issue business certificates through its API.

The certgen tool remains responsible only for bootstrapping local PKI for HTTPS and mTLS.

---

# Preserve Existing Functionality

This refactoring should not change the runtime behavior.

Only reorganize the repository and prepare it for future development.

Explain each structural change before applying it.
